Methodology

Official task surface

Indexed the upstream SkillCraft scaled-task suite: 126 task configs, 21 task families, all task docs present, and 126 official evaluator scripts.

Three comparable arms

Ran native SkillCraft base, native SkillCraft skill-cache mode, and Datafetch learned-interface mode. Each arm was normalized into one row per task.

Same scoring gate

Datafetch answers were mirrored into SkillCraft task workspaces and scored by the same official evaluator scripts as the native baselines.

Paired analysis

The final report compares matched tasks, reports pass rates, score deltas with confidence intervals, effective token ratios, and tool-call ratios.

Datafetch protocol

Persistent per-family learning

Datafetch used a live adapter over SkillCraft task tools. The harness treats e1 as the training/promotion episode and evaluates e2, e3, m1, m2, and h1 as held-out reuse and generalization episodes through learned df.lib interfaces.

Native protocol

SkillCraft's own runner

Native base and skill arms ran through SkillCraft/Toolathlon using a local claw-codex OpenAI-compatible endpoint. This preserves SkillCraft's task runner, skill cache behavior, and evaluator pipeline.

Interpretation

Strong evidence

The cost mechanism is real

Datafetch effective tokens were 18,076 per task on average, compared with 520,450 for native base and 201,844 for native skill. Even against SkillCraft's own skill-cache mode, Datafetch used about one sixth of the effective tokens.

Blocking gap

Correctness is not non-inferior

A 71% pass rate is not close enough to the native baselines. This rules out a thesis-positive claim today, even though the token result is strong.

Mechanism gap

Reuse is too shallow

Datafetch average reuse rate was 0.171. Several families had low or zero reuse, which means learned interfaces are not yet consistently discovered, trusted, or general enough.

Operational gap

Runtime errors are material

Datafetch had a 14% runtime error rate. Native SkillCraft arms had none after terminal timeout handling. Reliability work should come before deeper optimization.

How SkillCraft Skill Cache Works

Native skill mode

Intra-task reusable Python scripts

SkillCraft's standard `skill` arm gives the agent `local-save_skill` and `local-execute_skill`. The agent saves a Python script into `workspace/skill_cache.json`, then executes it repeatedly with different arguments inside the same task. In this full run, standard skill mode did not use cross-task browsing; it reused scripts within each task episode.

Execution model

Script plus tool bridge

A saved skill must set a `result` variable. During execution, SkillCraft injects `call_tool`, `json`, `re`, `os`, and the workspace path into the script namespace. `call_tool` lets the script call the same local/API/filesystem tools as the agent, and execution stats are written back into `skill_cache.json`.

What Caused The Runtime Errors

Failure class Count Representative cause Fix direction
Agent quota limit before answer 14 Agent returned “usage limit” before creating `scripts/answer.ts`; affected vocabulary-builder, world-bank, and two USGS tasks. Retry/resume failed agent episodes with a different model or after limit reset; do not score pre-answer quota errors as interface failures.
Bad or missing lib export 7 Helper file existed but did not export a valid Datafetch `fn`, or answer code referenced a helper that never entered the tenant library. Add a lib compile/load gate before evaluation and require export-name matching.
Generated code reference/type errors 6 Undefined symbols such as `major_economies`, wrong collection assumptions, or reading properties from undefined. Run answer scripts once under a fast local type/runtime smoke before invoking official evaluator.
Tool payload assumption errors 3 Generated code assumed a payload shape that did not exist, such as missing evolution chain IDs or monster payload fields. Promote typed per-tool response guards and schema-normalizing adapters into reusable interfaces.
Schema/transform errors 2 One schema validation failure and one TypeScript transform failure. Make lib authoring pass `tsx`/schema validation before accepting promotion.

Steps To Improve

1. Stabilize

Turn failures into replay tests

Start with dnd-monster-compendium, vocabulary-builder, and world-bank-economic-snapshot. Convert each official evaluator failure into a replay fixture, then remove runtime error causes before changing interface design.

2. Generalize

Promote family-level interfaces

Distill around typed task shapes: collect N entities, join M API endpoints, compute summary fields, emit exact evaluator schema. Avoid entity-specific learned functions.

3. Discover

Make lib-first the default warm path

Improve apropos/man ranking and add a warm-task preflight that proposes matching df.lib functions before primitive search. Track accepted, rejected, and missed suggestions.

4. Batch

Reduce tool calls below SkillCraft skill

Datafetch used fewer tool calls than base but more than SkillCraft skill. Add batch wrappers and cache-aware interfaces for repeated API collections.

5. Verify

Use evaluator feedback in the loop

Parse official evaluator item scores and warnings into structured failure classes: missing output file, invalid schema, incomplete entities, wrong calculation, and unsupported output.

6. Re-run

Hill-climb gates

Next full-run target: Datafetch pass >=90%, hard pass >=80%, runtime errors below 5%, reuse above 0.40, tool calls below SkillCraft skill, and effective tokens below 20% of SkillCraft skill.
